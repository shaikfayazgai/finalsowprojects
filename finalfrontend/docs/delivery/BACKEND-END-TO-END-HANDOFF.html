0) Final v2 Workflow (Source of Truth)

This is the authoritative end-to-end flow the product is built to. It supersedes the earlier two-gate description. Responsibility split is locked: Mentor is assigned by Glimmora; Reviewer is assigned by Enterprise.

0.1 The 12-step flow

  1. Enterprise admin uploads a SOW.
  2. Enterprise internal gates approve in order: Finance → Security → Legal.
  3. Glimmora platform performs the final approval gate, and Glimmora assigns the Mentor.
  4. Enterprise assigns the Reviewer (acceptance authority).
  5. Enterprise decomposes the approved SOW into tasks.
  6. Tasks are published to skilled contributors with the price visible first.
  7. Contributors review the price and click “I’m interested”.
  8. Enterprise reviews the interested pool and selects ONE contributor; others are notified the task is closed.
  9. Selected contributor works and submits.
  10. Mentor (Glimmora) reviews first: approve or request rework.
  11. Enterprise Reviewer performs final acceptance of the milestone.
  12. Enterprise pays the milestone (base + GST + total); contributor payout is released.
flowchart TD U["Enterprise admin uploads SOW"] --> FIN["Finance gate (enterprise)"] FIN --> SEC["Security gate (enterprise)"] SEC --> LEG["Legal gate (enterprise)"] LEG --> PLAT["Glimmora platform approval"] PLAT --> MENT["Glimmora assigns Mentor"] MENT --> REV["Enterprise assigns Reviewer"] REV --> DEC["Enterprise decomposes into tasks"] DEC --> PUB["Publish tasks (price visible)"] PUB --> INT["Contributors show interest"] INT --> SEL["Enterprise selects ONE"] SEL --> WORK["Contributor works + submits"] WORK --> MGATE["Mentor review (Glimmora)"] MGATE --> EGATE["Enterprise Reviewer acceptance"] EGATE --> PAY["Milestone payment + payout"]

0.2 SOW approval state machine

Enterprise internal gates run first; the Glimmora platform gate is last.

SOW.status: draft | approval | approved | rejected | withdrawn | archived
SOW.stage (when status = approval): finance -> security -> legal -> platform

draft --submit--> approval@finance
approval@finance  --approve--> approval@security   (role: ent.finance / admin)
approval@security --approve--> approval@legal      (role: ent.it / admin)
approval@legal    --approve--> approval@platform   (role: ent.compliance / admin)
approval@platform --approve--> approved            (Glimmora platform only)
any enterprise gate --reject--> rejected
any enterprise gate --send_back--> earlier stage
platform --send_back--> draft
Four-eyes: SOW owner cannot approve their own SOW's enterprise gates.

0.3 Task marketplace state machine

Task (marketplace): open -> assigned
Per-contributor interest: interested -> selected | not_selected | withdrawn

open --publish--> visible to skilled pool (price shown)
contributor --express interest--> interest=interested
enterprise --select one--> winner.interest=selected,
                           others.interest=not_selected,
                           task.status=assigned
selected contributor --> downstream delivery (assigned -> in_progress -> submitted)

0.4 Delivery + acceptance gates

submitted --Mentor (Glimmora) approve--> mentor_approved
mentor_approved --Enterprise Reviewer accept--> enterprise_accepted (milestone payable)
enterprise_accepted --pay milestone--> paid -> contributor payout released
Mentor or Reviewer may request rework -> back to contributor.
Payment is gated on Enterprise Reviewer acceptance, not mentor approval alone.

0.5 Responsibility split (locked)

  • Mentor — assigned by Glimmora (platform), at the platform approval gate. Frontend: /admin/sow/[sowId] mentor panel; persisted in sow-staffing overlay.
  • Reviewer — assigned by Enterprise (admin/PMO) from licensed reviewer members. Frontend: /enterprise/sow/[sowId] staffing section.

1) What This Doc Is

This HTML is a full backend handoff map for GlimmoraTeam. It explains how each portal works, how each persona works, how personas and portals connect, and how the end-to-end journey moves from SOW intake to project delivery and payment.

Target audience: backend engineers implementing or validating /api/v1/* behavior and state transitions.

2) System Foundation

2.1 High-Level Runtime Topology

flowchart LR A[Browser: portal users] --> B[Next.js Frontend] B --> C[Frontend API routes /api/*] C --> D[Backend Service /api/v1/*] D --> E[(PostgreSQL)] B --> F[NextAuth session] F --> C B --> G[Client RBAC gate]

2.2 Practical Architecture Notes

  • Frontend keeps public API shape at /api/*; backend implements internal contract at /api/v1/*.
  • Current codebase keeps NextAuth and many mocks in frontend; backend scaffold is decoupled and should own stable business APIs.
  • Enterprise RBAC is enforced client-side for navigation and route access; backend must still enforce server-side authorization for all mutations.
  • SOW approval pipeline is two-gate in Phase 1: commercial then final.

4) Persona Map (How Each Persona Works)

Persona Main Responsibilities Can Write Cannot Do (Key Guardrails)
Super Admin (Platform) Tenant provisioning, platform commercial review, governance controls. Commercial gate decisions, tenant setup. Should not act as enterprise final approver.
Enterprise Sponsor Create/submit SOWs; track pipeline and delivery. SOW intake and submission. Cannot final-approve own SOW (four-eyes).
Enterprise Admin Tenant enterprise authority and final sign-off. Final approval on enterprise gate; tenant controls. Should not be same actor as SOW owner for final gate.
PMO Decompose approved SOWs, run project delivery. Decomposition and delivery-level actions. Cannot access SOW intake authoring flow.
Reviewer Acceptance decisions on submitted work. Queue actions, approve/reject/rework outcomes. No SOW origination or tenant admin scope.
Contributor Accept tasks, submit deliverables, receive payouts. Task acceptance and submission artifacts. No enterprise admin or commercial actions.
Mentor (base / lead) Mentorship and review support. Queue/session actions; coaching notes. Escalation is limited by mentor role tier.

7) End-to-End Core Flow: SOW to Project

7.1 Canonical Phase 1 Golden Path

  1. Sponsor creates SOW from upload/author and submits for approval.
  2. Platform Admin reviews and approves commercial gate.
  3. Enterprise Admin (non-owner) approves final gate.
  4. PMO decomposes approved SOW into plan and tasks.
  5. Enterprise Admin approves plan.
  6. Project is provisioned with starter tasks for delivery.

7.2 Sequence Diagram (Detailed)

sequenceDiagram participant Sponsor participant EntUI as Enterprise UI participant API as /api + /api/v1 participant Admin as Platform Admin participant EntAdmin as Enterprise Admin participant PMO Sponsor->>EntUI: Create + submit SOW EntUI->>API: POST /sow (draft + payload) EntUI->>API: submitSow(id) API-->>EntUI: status=approval, stage=commercial Admin->>API: approve commercial gate API-->>EntUI: stage=final EntAdmin->>API: approve final gate API-->>EntUI: status=approved PMO->>API: create decomposition plan API-->>PMO: planId EntAdmin->>API: approve plan API-->>PMO: project provisioned

7.3 Invariants Backend Must Preserve

  • Invariant A: stage order strictly follows commercial -> final.
  • Invariant B: sponsor cannot finalize own SOW.
  • Invariant C: decomposition allowed only after SOW is approved.
  • Invariant D: plan approval should fail fast if plan payload is invalid/unlinked.

8) Delivery and Payment End-to-End

8.1 Execution Loop

  1. PMO/enterprise assigns task to contributor.
  2. Contributor accepts task and submits artifact.
  3. Reviewer/acceptance actor approves or requests rework.
  4. Once milestone is accepted, payment release action is enabled.
  5. Payout record is generated/released and contributor sees earnings update.

8.2 Payment Logic Clarification (Important)

  • Assign task does not mean paid; it only moves assignment state.
  • Pay milestone is finance release action after acceptance checkpoint.
  • Payouts are ledger-level records, not the same as assignment status.
  • UI now includes payment confirmation details (amount/GST/total) before final release click.
flowchart TD T1[Task Assigned] --> T2[Contributor Accepts] T2 --> T3[Contributor Submits] T3 --> T4{Review decision} T4 -->|Rework| T2 T4 -->|Accepted| T5[Milestone accepted] T5 --> T6[Pay milestone action] T6 --> T7[Payout record released]

11) State and Data Model Reference

11.1 SOW Core Status Model (service-layer type)

draft -> approval -> approved/rejected -> withdrawn/archived

Stages: commercial, final.

11.2 Cross-Domain Entities You Should Model Cleanly

  • Tenant: license scope, member roles, policy context.
  • User + Role grants: normalized role codes (ent.admin style).
  • SOW: versioned payload, confidentiality, approvals timeline.
  • Plan: decomposition output and approval state.
  • Project + Task: execution state and assignment ownership.
  • Submission + Review: quality loop and decision rationale.
  • Payout: amount computation, release state, external references.
  • Audit: non-repudiable event history.

13) Backend Team Implementation Checklist

13.1 Must-Haves

  • Implement all scoped /api/v1/* contract endpoints used by frontend proxies.
  • Enforce server-side RBAC for every write endpoint and sensitive reads.
  • Implement two-gate approval transition guards with four-eyes final sign-off rule.
  • Treat confidentiality as query-level access constraint, not only UI metadata.
  • Use idempotency keys for payout and webhook-sensitive actions.
  • Emit audit events for all stage transitions and financial mutations.

13.2 Recommended Early Integration Order

  1. Auth + /me role claims.
  2. SOW create/list/detail/update + submit + stage decisions.
  3. Decomposition/project provisioning events.
  4. Task assignment + submission + acceptance queue.
  5. Payout release and payment rails.

14.2 Deep Understanding Summary (One-Page Mental Model)

GlimmoraTeam is a controlled handoff pipeline: business intent becomes SOW, SOW becomes approved contract, approved contract becomes executable plan, plan becomes project tasks, tasks become submissions, submissions become acceptance decisions, and accepted work becomes payout records. Every stage is role-scoped, auditable, and policy-gated.
