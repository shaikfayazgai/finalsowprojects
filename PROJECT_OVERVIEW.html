1Architecture — how a request travels

The browser only ever talks to the frontend. The frontend's API proxy attaches the user's token and forwards to the gateway, which routes to the right role backend by URL prefix. All five backends share one database.

Browser  ─▶  Next.js frontend (:3000)
                 │  every page calls /api/**/route.ts  (the proxy layer)
                 ▼
        proxy: reads NextAuth session → attaches  Authorization: Bearer <JWT>
                 ▼
            GATEWAY (:9000)   ── longest URL-prefix wins ──▶  one role backend
                 │
   /api/mentor → 8101   /api/superadmin,/api/admin → 8102   /api/enterprise,/api/v1/sow → 8103
   /api/contributor → 8104   /api/reviewer → 8105
                 ▼
       ONE shared Neon Postgres "glimmora"   +   MongoDB (audit log)
Why it works: every backend signs & verifies JWTs with the same API_SECRET_KEY, so one login token is valid across all five services — the gateway just forwards it. They coordinate through shared tables (decomp_tasks, contributor_tasks, mentor_reviews, reviewer_assignments, payouts).

4How to run it

Start order: 5 backends → gateway → frontend. Each backend is a single-worker uvicorn started by a small launcher.

Prerequisites

  • Python 3.12 (C:\Program Files\Python312\python.exe) with each backend's deps installed
  • Node + npm for the frontend
  • The .env files present in each backend dir + newfrontend/frontend/.env.local (Neon DB URL, API_SECRET_KEY, Mongo URI)
  • Critical: the frontend's AUTH_SECRET must equal the backends' API_SECRET_KEY (so JWTs verify)

Start everything

# paths
PY="C:/Program Files/Python312/python.exe"
R="C:/tmp/run_backend.py"
B="E:/GLIMMORA/educore/GTPROJECT/backends"

# 1) the five role backends
"$PY" "$R" "$B/mentor/backend"      8101 app:app  &
"$PY" "$R" "$B/super-admin/backend" 8102 app:app  &
"$PY" "$R" "$B/enterprise/backend"  8103 app:app  &
"$PY" "$R" "$B/freelancer/backend"  8104 app:app  &
"$PY" "$R" "$B/reviewer/backend"    8105 app:app  &

# 2) the gateway (run from the backends/ dir)
cd "$B" && "$PY" -m uvicorn gateway:app --host 127.0.0.1 --port 9000  &

# 3) the frontend
cd "E:/GLIMMORA/educore/GTPROJECT/newfrontend/frontend" && npm run dev

Verify it's up

# all 7 ports listening?
netstat -ano | grep LISTENING | grep -E ":(3000|8101|8102|8103|8104|8105|9000) "
# frontend responds?
curl http://localhost:3000        # → 200
Open http://localhost:3000 → the marketing home page means the system is ready.

7Pricing model (the money)

The contributor is paid in full. Glimmora's margin and GST are added on top to form the enterprise price. The golden rule: enterprise / mentor / reviewer never see the contributor's pay.

The formula

contributor pay   = what Glimmora sets per task
client price      = contributor ÷ (1 − commission%)
GST               = client price × gst%
enterprise pays   = client price + GST

Example: pay ₹7,000 · commission 15% · GST 18% → client ₹8,235 → +GST ₹1,482 → enterprise pays ₹9,718.

MentorTask brief only — no money

ContributorOwn payout only (paid in full)

8Operational notes

  • Backend changes need that one port restarted (single-worker, no auto-reload). The frontend hot-reloads.
  • Database resilience is built in — TCP keepalives + reconnect-retry on the Neon connection, so the "everything went blank / 0 tasks" idle-drop won't recur. If a backend ever wedges, restart just that port.
  • Payments are simulated (Razorpay test-mode planned) — no real money moves.
  • Shared DB: all five backends read/write the same Neon "glimmora" database, so the flow stays consistent across roles.
Secrets: the .env files hold the live Neon DB string + API_SECRET_KEY. Keep the repo private and rotate them if they're ever exposed.
